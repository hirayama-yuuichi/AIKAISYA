# 仮想API作成会社 — ブラッシュアップ案

---

## コンセプト

AIエージェントが各部門を担当し、**トレンド調査 → 設計 → 製造 → 検証 → 発信** のサイクルを自律的に回す「仮想API開発会社」をアプリケーションとして実装する。

全部門に **トレンド確認係** と **API使用量確認係** を常駐させ、技術の鮮度とコストを常時監視する。

---

## 組織レイヤー構造

### Layer 1 — 経営・戦略（Head）
> 会社の方向性を決め、組織そのものを進化させる

| 部門・課 | 役割 | 主な出力 |
|---|---|---|
| 会社将来方針決定課 | 長期ビジョン策定 | 中長期ロードマップ |
| 会社方針実現課 | ビジョンを具体的指示に変換 | 実行計画書 |
| 総合判定部門 | 全部門報告を統合し最終承認 | GO/HOLD/NO-GO判定 |
| 新規部門立案部門 | 組織の欠陥・拡張機会の特定 | 新部門設立提案書 |
| 新規部門判定部門 | 新部門設立のGO/NO-GO審査 | 設立承認・否決レポート |

### Layer 2 — ガバナンス（Spine）
> 全部門が従うルールを作り、知識を蓄え、伝達する

| 部門 | 役割 | 主な出力 |
|---|---|---|
| 共通方針部門 | 全社標準・ガイドライン策定 | 共通方針書（憲法） |
| 方針蓄積部門 | 成功・失敗事例のナレッジDB化 | ナレッジベース |
| 方針共有部門 | 各部門へのリアルタイム同期 | 方針更新通知 |
| ドキュメント用語解説部門 | 専門用語の翻訳・社内辞書管理 | 用語辞書 |

### Layer 3 — 設計・製造（Hand）
> 実際のAPIを企画・設計・実装・検証する

| 部門 | 役割 | 主な出力 |
|---|---|---|
| トレンド確認部門 | 市場・技術動向のスキャン | トレンドレポート |
| 新技術調整部門 | 新技術の導入可否判断 | 技術採用判定書 |
| ドキュメント作成部門 | API仕様書（OpenAPI等）作成 | 仕様書 |
| 資料作成部門 | 社内向け報告資料作成 | 各種レポート |
| 製造工程分解部門 | 設計図をタスク（WBS）に分解 | タスクリスト |
| 製造工程管理部門 | 進捗監視・ボトルネック解消 | 進捗レポート |
| セキュリティー確認部門 | 脆弱性診断・認証設計レビュー | セキュリティ診断結果 |
| 運用確認部門 | デプロイ後のリソース消費予測 | 負荷・コスト試算 |
| 確認部門 | 各工程の品質確認 | 品質チェックレポート |

### Layer 4 — 成果・発信（Face）
> 成果を数値化し、外部へ発信する

| 部門 | 役割 | 主な出力 |
|---|---|---|
| 使用料部門 | API利用料金モデルの算出 | 料金シミュレーション |
| 業績発表部門 | 収益・ROIの可視化と報告 | 業績レポート |
| 社外アピール部門 | 技術広報・マーケティング | 技術ブログ・SNS・PR文 |

---

## 全部門共通の配置係

```
各部門
  ├── トレンド確認係     ← その部門の文脈でトレンドを監視
  └── API使用量確認係   ← その部門の文脈でコストを監視
```

---

## エージェントの指示体系（3層）

```
Layer A: 固有ミッション    「あなたはセキュリティー確認部門です」
Layer B: 共通方針（憲法）  「認証はOAuth2.0必須、命名規則はキャメルケース」
Layer C: 動的方針共有      「現在、GraphQL導入が決定されました」
```

---

## 情報フロー（メインパス）

```
会社将来方針決定課
  └─▶ 会社方針実現課
        └─▶ トレンド確認部門 ─▶ 新技術調整部門
                                    └─▶ ドキュメント作成部門
                                          └─▶ 製造工程分解部門
                                                └─▶ 製造工程管理部門
                                                      ├─▶ セキュリティー確認部門
                                                      ├─▶ 運用確認部門
                                                      └─▶ 確認部門
                                                            └─▶ 総合判定部門
                                                                  ├─▶ 使用料部門
                                                                  ├─▶ 業績発表部門
                                                                  └─▶ 社外アピール部門
```

横断サポート（常時稼働）：共通方針部門 / 方針蓄積部門 / 方針共有部門 / ドキュメント用語解説部門

## 実装方針

| 項目 | 内容 |
|---|---|
| 言語 | Python |
| フレームワーク候補 | LangGraph（ステート管理・フロー制御） |
| エージェントベース | Claude API（各部門のSystem Instruction） |
| 状態管理 | 共有Stateオブジェクトで部門間データを受け渡し |
| 方針管理 | `config/policy.yaml` に共通方針を定義 |

---

## スモールスタート計画

| フェーズ | 内容 | 成果物 |
|---|---|---|
| Phase 1 | State定義・共通方針読み込み | `core/state.py`, `core/policy.py` |
| Phase 2 | 基底エージェントクラス作成 | `agents/base_agent.py` |
| Phase 3 | トレンド確認部門を単独で動かす | `agents/trend/` |
| Phase 4 | ドキュメント作成部門を追加し2部門連携 | `agents/document/` |
| Phase 5 | セキュリティー確認部門を追加し3部門ループ | `agents/security/` |
| Phase 6 | 総合判定部門でGO/NO-GO判定 | `agents/governance/` |
